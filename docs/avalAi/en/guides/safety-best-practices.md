# Safety Best Practices

Developing applications with powerful AI models requires careful consideration of safety and security. Implementing robust safety measures helps protect both your users and your application's integrity. Here are key best practices to follow when using the AvalAI API:

## Use Security Guardrails

- **Enable Automatic Detection of Sensitive Information:** Utilize AvalAI's Guardrails system to automatically detect and redact sensitive information like API keys, tokens, and credentials from your API calls before they reach the model.
- **Scanning Scope:** Guardrails only scan specific API endpoints and payload content:
  - **Supported Endpoints:** `v1/chat/completions`, `v1/completions`, `v1/messages`, `v1/responses`, `v1beta/models` (Gemini API)
  - **Scanned Content:** Only the body payload fields `"messages"`, `"input"`, and `"prompt"` are analyzed
  - **Size Limitations:** Content larger than 10MB will be ignored by the guardrail system and will not be scanned
- **Activation Options:** Enable Guardrails either at the user level in your settings or on a per-request basis by including `guardrails: ["hide-secrets"]` in your API request data.
- **Implementation Example:**

```json
{
  "messages": [
    {
      "role": "user",
      "content": "Here's my question about the API..."
    }
  ],
  "guardrails": [
    "hide-secrets"
  ]
}
```
- **Multiple Layers of Protection:** While Guardrails provides significant protection, always follow security best practices and avoid sending sensitive information in your API calls whenever possible.

## Use Moderation Tools

- **Leverage AvalAI's Moderation Endpoints:** Utilize AvalAI's [moderation capabilities](/api-reference/moderation.md) to detect and filter potentially harmful or inappropriate content generated by the models or submitted by users. This is a crucial first line of defense.
- **Develop Custom Filters:** Depending on your specific use case, consider developing your own content filtration system tailored to your application's needs, potentially in addition to AvalAI's moderation.

## Adversarial Testing ("Red Teaming")

- **Test Robustness:** Actively try to "break" your application by testing it with a wide range of inputs and user behaviors. Include both typical use cases and adversarial attempts.
- **Identify Vulnerabilities:** Check if the model can be easily led off-topic or manipulated through prompt injections (e.g., "ignore previous instructions and do X"). Address any weaknesses found.

## Implement Human Oversight (Human-in-the-Loop)

- **Review Critical Outputs:** Wherever feasible, especially in high-stakes domains (like medical advice, financial decisions, or code generation), ensure a human reviews the model's output before it's used or presented to the end-user.
- **Provide Context for Reviewers:** Humans involved in the review process should understand the model's limitations and have easy access to any source information needed to verify the output's accuracy.

## Employ Effective Prompt Engineering

- **Constrain Output:** Use clear and specific instructions in your prompts ([prompt engineering](/guides/prompt-engineering.md)) to guide the model's topic, tone, and style. This reduces the likelihood of generating undesirable content.
- **Provide Examples (Few-Shot Learning):** Include high-quality examples of the desired input-output behavior within the prompt to steer the model more effectively.

## Know Your Customer (KYC)

- **User Authentication:** Require users to register and log in to access your service. This creates accountability.
- **Stronger Verification (Optional):** Consider linking accounts to established providers (e.g., Google, social logins) or requiring payment information/ID verification for higher-risk applications, where appropriate.

## Constrain Inputs and Outputs

- **Limit Input Length:** Restricting the length of user input can help mitigate prompt injection attacks.
- **Limit Output Tokens:** Capping the maximum number of tokens in the response can prevent overly long or potentially abusive outputs.
- **Use Structured Inputs:** Whenever possible, use validated inputs like dropdown menus or selections from trusted lists instead of open-ended text fields.
- **Prefer Retrieval over Pure Generation:** For tasks like answering support queries, consider retrieving the best-matching existing article or information from a trusted backend source rather than generating a completely novel response from scratch.

## Allow Users to Report Issues

- **Provide Reporting Mechanism:** Offer a clear and easily accessible way for users to report problematic outputs, bugs, or other concerns (e.g., a dedicated email address, feedback form, reporting button).
- **Monitor and Respond:** Ensure these reports are monitored by humans and addressed appropriately.

## Understand and Communicate Limitations

- **Acknowledge Imperfections:** Be aware that large language models can sometimes generate inaccurate information ("hallucinate"), produce biased or offensive content, or fail in unexpected ways.
- **Assess Fitness for Purpose:** Evaluate whether the model is suitable for your specific use case and test its performance across diverse inputs.
- **Set User Expectations:** Clearly communicate the application's capabilities and limitations to your users to manage their expectations appropriately.

## Send End-User IDs for Monitoring

Sending unique, anonymized end-user IDs with your API requests helps AvalAI monitor for potential misuse and policy violations, allowing us to provide more actionable feedback if issues arise.

- **Anonymization:** Hash usernames or email addresses to avoid sending personally identifiable information (PII). Use session IDs for non-logged-in users.
- **Implementation:** Include the ID in the `user` parameter of your API request.

<summary>Responses API version This version uses `gpt-5.5` because `claude-haiku-4-5` may not be enabled for `/v1/responses` in the current AvalAI model data.</summary>

Use this version when the selected model supports `/v1/responses`. `messages` moves to `input`, and the final text is read from `response.output_text`.

```language-selector
python=:import os
from openai import OpenAI

client = OpenAI(
    api_key=os.environ["AVALAI_API_KEY"],
    base_url="https://api.avalai.ir/v1",
)

response = client.responses.create(
    model="gpt-5.5",
    instructions="You are a helpful assistant.",
    input="This is a safety test.",
)

print(response.output_text)

javascript=:import OpenAI from "openai";

const client = new OpenAI({
  apiKey: process.env.AVALAI_API_KEY,
  baseURL: "https://api.avalai.ir/v1",
});

const response = await client.responses.create({
  model: "gpt-5.5",
  instructions: "You are a helpful assistant.",
  input: "This is a safety test.",
});

console.log(response.output_text);

bash=:curl https://api.avalai.ir/v1/responses \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $AVALAI_API_KEY" \
  -d '
  {
    "model": "gpt-5.5",
    "input": "This is a safety test.",
    "instructions": "You are a helpful assistant."
  }'

```

- `messages` → `input`
- system message → `instructions` or a `developer` item
- `choices[0].message.content` → `response.output_text`
- for tools and multimodal output, inspect `response.output` by item `type`.
